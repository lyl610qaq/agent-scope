Runtime and indexing flow

Two separate paths: chat-time retrieval for answering, and manual reindex for syncing local knowledge into pgvector.

Chat request flow
1. User sends
message + conversationId
2. Retrieve memory
short-term recent turns + long-term low-risk facts
3. Retrieve RAG
question -> SiliconFlow embedding -> pgvector topK search
4. Assemble prompt
system prompt + memory + RAG + user message
5. Call agent
ReActAgent / configured chat model
6. Update memory
append short-term turn; extract low-risk long-term memory
Manual knowledge reindex flow
1. Trigger
POST /api/knowledge/reindex
2. Scan files
data/knowledge .md and .txt
3. Chunk
normalize text and split into stable chunks
4. Embed
SiliconFlow Qwen/Qwen3-Embedding-4B, dimensions 1024
5. Store
upsert source, content, checksum, metadata, vector(1024)
6. Report
indexed count, skipped count, failures

Important boundary

Chat requests never rescan files. They only query already-indexed pgvector rows. Reindex is explicit because it calls an external embedding API.
